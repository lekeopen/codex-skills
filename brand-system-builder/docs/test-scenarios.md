# Brand System Builder Test Scenarios

适用范围：仅设计 Brand System Builder 的验证场景，不涉及 `SKILL.md`、Skill 实现、自动化测试代码或品牌资产制作。

## 1. 测试目标

本文档验证 Brand System Builder 是否能够：

- 正确识别 `N`、`U`、`W`、`U+W` 四类路径。
- 根据项目现状裁剪阶段，而不是机械执行完整流程。
- 区分品牌建设、品牌升级和网站接入。
- 正确处理 `Required`、`Conditional`、`Reuse`、`Skip`。
- 在关键决策点停止并触发 Approval Gate。
- 未经批准时不把探索结果推进为正式品牌规范。
- 记录复用、跳过、回退和变更的依据。
- 最终形成可追溯、可治理的正式交付结果。

## 2. 场景覆盖矩阵

| ID | 案例 | 路径 | 主要验证重点 |
|---|---|---|---|
| TS-01 | 全新的个人品牌 | N | 从零建立战略与品牌系统 |
| TS-02 | 乐可开源 | U（轻量升级） | 已有产品但品牌体系不成熟 |
| TS-03 | 已有品牌只接入网站 | W | 复用品牌系统并完成网站映射 |
| TS-04 | 边大夫口腔 | U+W | 品牌升级与现有网站迁移协同 |

# TS-01：全新的个人品牌

## 背景（Background）

一位独立技术教育创作者准备建立个人品牌，计划通过个人网站、公众号、视频平台和演讲活动持续输出内容。

目前只有个人姓名、专业经历和少量文章，没有正式品牌定位、Logo、色彩、字体、语言风格或品牌规范。

## 用户目标（Goal）

从零建立一套真实可用的个人品牌系统，使不同内容渠道保持统一识别，同时避免把个人品牌设计成缺乏真实性的商业公司品牌。

## 输入材料（Inputs）

- 个人简介和专业经历
- 主要内容主题
- 代表性文章或视频
- 目标受众初步描述
- 未来使用渠道
- 用户喜欢和排斥的视觉参考
- 个人姓名、英文名或候选品牌名
- 隐私、肖像和个人表达边界
- 项目时间与预算限制

特意保留的不完整输入：

- 没有正式定位陈述
- 没有成熟品牌资产
- 用户可能只提供“简洁、高级、有温度”等模糊偏好

用于验证 Skill 是否先澄清战略，而不是直接生成 Logo 或配色。

## 预期 Workflow 路径

路径判定：`N`

```text
00 启动与路径判定
→ 01 发现与审计
→ 02 品牌战略基础
→ 03 品牌创意方向
→ 04 核心品牌系统
→ 05 应用与触点规则
→ 07 品牌规范整合
→ 08 发布规划（Conditional）
→ 09 交付与治理
```

阶段 06 网站品牌接入：

- 若本轮只建立品牌规范：`Skip`，记录网站尚未进入实施范围。
- 若同时定义新网站的品牌应用原则：`Conditional`。
- 不应误判为 `W`，因为不存在需要接入的现有网站。

阶段 01 应聚焦个人现状、内容、声誉和已有表达审计，而不是假设必须存在旧视觉资产。

## 必须触发的 Approval Gates

- `G0`：批准 `N` 路径、范围、渠道和成功标准。
- `G2`：批准目标受众、定位、品牌属性、承诺和语气。
- `G3`：批准一个创意方向。
- `G4`：批准核心品牌系统。
- `G5`：批准网站首页、内容封面、头像等高频触点规则。
- `G7`：批准正式品牌规范版本。
- `G8`：若包含公开发布计划，批准上线次序。
- `G9`：确认正式交付、资产所有权和维护方式。

`G1` 不要求批准旧资产的保留或替换，但应确认现状事实。

## 成功标准

- 正确判定为 `N`，不因已有文章或个人头像误判为 `U`。
- 在 `G2` 前不直接确定最终视觉系统。
- 战略能够回答“为谁、提供什么价值、为何值得关注”。
- 创意方向不是只有颜色或字体不同的表面方案。
- 个人表达、专业可信度与商业传播之间具有明确边界。
- 核心系统可覆盖头像、内容封面、个人网站和演讲材料。
- 网站阶段是否进入范围有明确记录。
- 所有正式资产都能追溯到已批准方向。
- 最终规范明确版本、使用范围和后续维护责任。

## 常见失败模式

- 收到“简洁、高级”等描述后直接生成 Logo。
- 将个人偏好错误等同于品牌战略。
- 套用企业品牌模板，忽略个人真实性和表达负担。
- 一次提供大量无实质差异的视觉方案。
- 未确认姓名、肖像或隐私边界。
- 跳过 `G2` 或 `G3`，直接进入核心系统。
- 为尚不存在的网站设计大量页面细节，造成范围扩张。
- 只交付视觉资产，没有语言规则和内容触点规则。
- 未确定用户本人如何维护未来的品牌资产。

# TS-02：乐可开源——已有产品，建立品牌系统

## 背景（Background）

乐可开源已经存在真实产品、代码仓库、用户认知和部分线上内容，但缺少完整、统一且经过批准的品牌系统。

现有名称和部分产品识别可能具有保留价值，但产品存在不代表已经具备成熟品牌战略。

## 用户目标（Goal）

在保留有效产品认知的基础上，建立适合开源项目长期发展的品牌定位、视觉语言和多触点规则，统一产品、代码仓库、文档、社区及官网表达。

## 输入材料（Inputs）

- 产品介绍和现有功能
- GitHub 或其他代码仓库
- README、文档站和发布说明
- 当前 Logo、图标、截图和界面
- 官网或产品页面（如存在）
- 用户、开发者和贡献者反馈
- 项目历史及命名来源
- 已有社区渠道
- 同类开源项目参考
- 必须保留的产品或技术约束
- 开源许可证及资产授权信息

故意设置的输入冲突：

- 团队认为这是“新品牌”。
- 产品实际上已有用户、名称和可识别资产。
- 部分旧资产受用户认可，但未形成规范。

用于验证阶段 00 是否会根据事实判断路径。

## 预期 Workflow 路径

路径判定：`U（轻量升级）`

```text
00 启动与路径判定
→ 01 发现与资产审计
→ 02 品牌战略基础
→ 03 品牌创意方向
→ 04 核心品牌系统
→ 05 应用与触点规则
→ 06 网站品牌接入（Conditional）
→ 07 品牌规范整合
→ 08 发布与迁移规划
→ 09 交付与治理
```

关键裁剪判断：

- 阶段 02 不能因已有产品而直接跳过。
- 阶段 01 必须区分产品事实、社区认知和正式品牌资产。
- 若官网已存在且本轮包含改造，路径应升级为 `U+W`。
- 若官网不在范围内，阶段 06 应明确标记 `Skip` 或后续项目。
- 产品功能设计不属于品牌系统构建，不应被带入本流程。

## 必须触发的 Approval Gates

- `G0`：批准轻量 `U` 路径、范围及网站是否纳入。
- `G1`：批准现有名称、Logo、产品识别和社区资产的保留／优化／替换边界。
- `G2`：批准新的品牌战略基础。
- `G3`：批准品牌创意方向。
- `G4`：批准核心品牌系统及旧资产变化范围。
- `G5`：批准代码仓库、文档、产品界面等高优先级触点规则。
- `G6`：仅在官网接入纳入本次范围时触发。
- `G7`：批准正式品牌规范版本。
- `G8`：批准旧资产迁移、共存和停用计划。
- `G9`：批准维护责任和社区贡献规则。

## 成功标准

- 不以用户自称“新品牌”作为唯一判定依据。
- 识别并保护已有产品认知和可复用资产。
- 清楚区分品牌系统问题与产品功能、交互或技术架构问题。
- 对每项旧资产给出保留、优化或替换结论及理由。
- 战略同时覆盖最终用户、开发者和社区贡献者。
- 品牌系统适用于 README、文档、版本发布、产品界面和社区传播。
- 若网站不在范围内，不生成网站实施方案。
- 若现有网站被纳入，能够重新判定为 `U+W` 并触发 `G6`。
- 新旧资产迁移不会导致仓库、产品和文档长期使用不同版本。
- 明确品牌资产授权及外部贡献的治理方式。

## 常见失败模式

- 因“没有品牌手册”便误判为完全从零的 `N`。
- 因“已有产品”便默认品牌战略已经成熟。
- 全部推翻现有资产，损害已有用户认知。
- 只设计 Logo，忽略文档、代码仓库和产品界面。
- 把产品 UI 重构纳入品牌项目，造成范围失控。
- 未检查旧 Logo、字体或素材的授权。
- 官网存在时自动进入网站改造，未取得范围批准。
- 未建立新旧版本共存和迁移规则。
- 将社区贡献内容直接视为正式品牌规范。

# TS-03：成熟品牌仅接入现有网站

## 背景（Background）

“青禾建筑事务所”是一个假设案例。该品牌已有经过批准且仍有效的品牌规范，包括 Logo、色彩、字体、摄影和语言规则。

公司网站已运行多年，但未正确采用品牌系统，页面组件、颜色、字体和内容语气不一致。用户明确要求保持品牌战略与核心品牌资产不变，本次只处理网站品牌接入。

## 用户目标（Goal）

将现有品牌系统准确映射到现有网站，在不重新设计品牌、不改变信息架构和核心产品功能的前提下，提高网站的一致性、可访问性和品牌辨识度。

## 输入材料（Inputs）

- 当前有效的品牌规范及版本号
- 正式 Logo、字体和其他品牌资产
- 资产授权证明
- 现有网站地址
- 网站代码库或设计文件
- 页面和组件清单
- 当前 Design Tokens 或 CSS 变量
- 内容样本
- 网站分析数据
- SEO 基线
- 可访问性和浏览器兼容要求
- 技术栈、CMS 与发布限制
- 明确的禁止变更项

故意设置的约束：

- 不允许修改 Logo。
- 不重新定义品牌战略。
- 不改变网站信息架构。
- 某品牌字体无法用于网页正文。

用于验证复用和网站约束冲突处理。

## 预期 Workflow 路径

路径判定：`W`

```text
00 启动与路径判定
→ 01 网站与品牌适配审计
→ 02 品牌战略基础（Reuse）
→ 03 品牌创意方向（Skip / Reuse）
→ 04 核心品牌系统（Reuse，必要时仅补充数字媒介规则）
→ 05 网站关键触点规则（Conditional）
→ 06 网站品牌接入
→ 07 品牌规范整合
→ 08 发布与迁移规划
→ 09 交付与治理
```

裁剪要求：

- 阶段 02、03、04 的复用来源和版本必须明确。
- 不得把 `Reuse` 当成无记录的跳过。
- 字体无法在网页使用时，应形成数字媒介适配或例外规则。
- 若适配需要改变核心品牌战略或核心资产，必须停止 `W` 路径并请求重新定界，而非擅自升级品牌。

## 必须触发的 Approval Gates

- `G0`：批准 `W` 路径、禁止变更项和网站接入范围。
- `G1`：批准网站审计及保留／调整／重构分类。
- `G2`：确认复用的品牌战略版本。
- `G4`：确认复用的核心品牌系统；如有数字媒介补充规则，必须批准。
- `G5`：批准网站高频触点和关键状态规则。
- `G6`：批准页面、组件、Token、内容及迁移影响范围。
- `G7`：批准包含网站适配规则的正式规范版本。
- `G8`：批准网站发布、回退和分批迁移计划。
- `G9`：确认交付及后续网站品牌治理责任。

`G3` 默认不重新开启；若需要新的创意方向，说明当前项目已超出纯 `W` 范围。

## 成功标准

- 正确判定为 `W`，没有发起品牌重塑。
- 明确记录复用的品牌规范版本。
- 对关键页面、组件、断点和交互状态完成影响分析。
- 品牌元素被映射到网站层级，而非只替换 Logo 和主色。
- 字体等媒介冲突通过受控适配规则解决。
- 品牌变化与 UX、信息架构、功能变化保持分离。
- SEO、可访问性、性能和兼容性基线不被忽略。
- `G6` 前不进入接入执行或发布计划。
- 发布计划包含回退条件。
- 最终规范清楚说明网站例外及其适用范围。

## 常见失败模式

- 把网站品牌接入误解为重新设计品牌。
- 未验证品牌规范的版本和资产授权。
- 只更换 Logo、颜色和字体。
- 未覆盖表单、错误、空状态、移动端和交互状态。
- 字体不可用时擅自替换，却没有例外规则或批准。
- 将页面结构或产品功能重构混入品牌接入。
- 忽略 SEO、可访问性、性能或浏览器兼容性。
- 未建立 Design Tokens 或等价映射关系。
- 未设计发布回退方案。
- 网站适配结果与正式品牌规范分离，形成第二套事实来源。

# TS-04：边大夫口腔——品牌升级并接入已有网站

## 背景（Background）

边大夫口腔已有名称、视觉资产、患者认知和已上线网站。当前品牌表达及网站呈现存在一致性、可信度或数字体验方面的问题，需要完成品牌升级并接入现有网站。

该项目涉及真实医疗服务场景，品牌变化必须保护患者信任、既有业务流程、内容准确性、搜索表现和网站稳定性。

## 用户目标（Goal）

保留已有品牌中具有认知价值的部分，建立更专业、可信、亲和的品牌系统，并安全地映射到现有网站及患者关键触点。

## 输入材料（Inputs）

- 当前网站及主要页面
- 网站代码、CMS 或设计文件
- 当前 Logo、字体、色彩和宣传材料
- 门诊环境与导视照片
- 医生介绍、项目介绍和患者教育内容
- 预约、咨询及联系方式流程
- 当前搜索排名和访问基线
- 用户或患者反馈
- 当前品牌的历史和认知资产
- 医疗内容审核要求
- 隐私、合规和广告表达限制
- 技术栈、第三方预约服务和发布约束
- 明确必须保留的名称、域名或业务流程

故意设置的风险条件：

- 旧 Logo 仍被部分患者识别。
- 网站存在历史 SEO 页面。
- 新品牌字体可能影响中文可读性和页面性能。
- 品牌语气升级不能演变为医疗效果承诺。
- 网站无法一次性全部迁移。

## 预期 Workflow 路径

路径判定：`U+W`

```text
00 启动与路径判定
→ 01 品牌、触点与网站审计
→ 02 品牌战略校准
→ 03 品牌创意方向
→ 04 核心品牌系统
→ 05 患者重点触点规则
→ 06 网站品牌接入
→ 07 品牌规范整合
→ 08 分阶段发布与迁移规划
→ 09 交付、治理与持续演进
```

关键依赖：

- `01 → 03`：未批准保留／优化／替换边界，不得进入创意方向定稿。
- `02 → 03`：专业、可信、亲和等属性必须先转化为可判断的战略原则。
- `04 → 06`：核心系统未批准，不得开始正式网站映射。
- `06 → 08`：页面和组件影响不清楚，不得制定上线批次。
- 若网站限制使核心品牌系统不可用，应回退到阶段 04 或 06，而不是临时修改正式规则。

## 必须触发的 Approval Gates

- `G0`：批准 `U+W` 路径、范围、不可变条件、合规边界和成功标准。
- `G1`：批准旧品牌、患者触点及网站资产的保留／优化／替换边界。
- `G2`：批准品牌战略变化和患者沟通原则。
- `G3`：批准一个品牌创意方向。
- `G4`：批准核心品牌系统及升级变化范围。
- `G5`：批准官网、预约入口、医生介绍和患者教育等高影响触点规则。
- `G6`：批准网站接入范围、组件影响、内容变化、技术约束和风险接受程度。
- `G7`：批准正式品牌规范版本。
- `G8`：批准分阶段发布、旧资产共存、SEO 保护和回退方案。
- `G9`：确认正式交付、品牌负责人和医疗内容审核责任。

## 成功标准

- 正确判定为 `U+W`，没有拆成彼此脱节的品牌设计和网站换肤。
- 审计同时覆盖品牌资产、网站、患者旅程及线下关键触点。
- 明确说明哪些旧资产保留、优化或替换及其原因。
- 新方向能够提升专业可信度，同时保持患者可理解和可接近性。
- 不产生未经证实的医疗效果承诺。
- 核心品牌系统支持中文内容、移动端和真实医疗信息密度。
- 网站映射覆盖关键页面、组件、状态及第三方服务。
- 品牌变化与网站功能变化被明确区分。
- 迁移计划保护 URL、索引、结构化数据和关键转化入口。
- 支持新旧资产阶段性共存，并明确停用条件。
- 若网站约束与品牌系统冲突，能够正确回退并重新批准。
- 最终规范、网站适配规则和治理责任形成唯一可信基准。

## 常见失败模式

- 只做品牌升级，不审计现有网站。
- 只做网站换肤，没有处理品牌战略和旧资产边界。
- 未经 `G1` 批准便替换具有患者认知的旧 Logo。
- 使用抽象审美词汇代替可验证的战略判断。
- 品牌语气产生夸大或不合规的医疗承诺。
- 忽略中文字体可读性、性能或授权。
- 只覆盖首页，遗漏预约、医生、项目、文章和错误状态。
- 品牌改造同时破坏 URL、SEO、结构化数据或预约流程。
- 网站限制出现后临时篡改品牌规则，未触发回退。
- 一次性切换所有资产，没有分批发布和回退方案。
- 未明确谁负责审核未来的品牌内容与医疗内容。

# 3. 跨场景验证要求

除各案例自身标准外，四个场景都应验证以下行为。

## 3.1 路径判定

系统不能只依赖用户使用的标签，而应根据实际状态判断：

- 没有成熟资产：`N`
- 已有认知或产品，需要建立或改变品牌系统：`U`
- 品牌系统有效，只接入现有网站：`W`
- 品牌系统变化且需改造现有网站：`U+W`

当范围变化时，应重新判定路径。

## 3.2 Approval Gate 阻断

每个强制 Gate 至少测试三种响应：

1. 明确批准：允许进入下一阶段。
2. 要求修改：保留当前阶段，不能将方案标记为正式。
3. 未明确表态：不得把沉默、模糊反馈或选择偏好解释为批准。

## 3.3 裁剪与复用

每个 `Reuse` 或 `Skip` 必须记录：

- 复用或跳过的阶段
- 理由
- 来源资产或规范版本
- 适用范围
- 对后续阶段的影响
- 是否经过必要确认

## 3.4 回退

至少验证以下回退：

- 战略发生变化：回到阶段 02。
- 创意方向无法支持关键触点：回到阶段 03。
- 核心规则在真实内容下冲突：回到阶段 04。
- 网站约束使接入不可行：回到阶段 04 或 06。
- 发布范围重大变化：回到阶段 08。

回退应只重开受影响决策，不得无理由重启整个项目。

## 3.5 范围控制

系统应拒绝或隔离以下范围扩张：

- 产品功能开发
- 网站技术实现
- 无关的 UX 或信息架构重构
- 营销活动执行
- 未经批准的大规模内容重写
- 未纳入范围的额外触点制作

可以记录为后续工作，但不能伪装成本阶段的必要产物。

# 4. 基线完整性标准

本测试场景基线可视为完整，当：

- 四条主路径均有独立场景覆盖。
- 四个指定案例全部纳入。
- 每个场景包含七项规定内容。
- Gate 与已批准的 Workflow v1 一致。
- 场景能够验证路径误判、裁剪错误、越权推进和范围扩张。
- `Reuse`、`Skip`、回退和变更控制均可被验证。
- 成功标准能够客观判断，而非仅描述“效果好”。
- 没有进入 `SKILL.md`、Skill 实现或测试代码设计。

## 通用性与案例隔离约束

1. “边大夫口腔”“乐可开源”“青禾建筑事务所”和“全新个人品牌”仅作为测试样例，不代表 Skill 支持范围的边界。

2. 具体项目名称、行业知识、品牌内容和测试结论只能存在于：

   - `tests`
   - `fixtures`
   - `examples`

3. 上述案例不得成为：

   - Skill 固定业务逻辑
   - 默认品牌模板
   - 行业专用流程
   - 项目名称判断条件
   - 隐式推荐方案
   - 项目专用分支

4. 正式 Skill 的核心流程只能保留通用规则：

   - `N / U / W / U+W` 路径判断
   - 阶段裁剪与依赖
   - Approval Gates
   - 回退与变更控制
   - 范围控制
   - `Required / Conditional / Reuse / Skip` 规则

5. 任何新项目都必须从阶段 00 开始，根据当次实际输入重新判断路径、范围、裁剪方式和成功标准。

6. 不得根据案例名称、行业相似性或历史测试结果，自动套用已有项目的路径、品牌方向、保留结论或行业假设。

7. 测试必须验证案例隔离：移除或替换具体项目名称后，核心 Workflow 的判断能力和行为应保持不变。
