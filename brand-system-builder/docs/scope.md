# Brand Lifecycle Scope

## 1. 核心定位

**Brand System Builder** 不应只是管理已有品牌规范，也不应以 Brand Guide 为起点。

它的真正职责是：

> **支持一个品牌从现状审计或零基础起步，经过品牌定义、身份创建、系统化、实际应用、技术接入、版本冻结，再进入长期维护与演进的完整生命周期。**

这里的“System Builder”包含两层含义：

- **Build the Brand**：建立品牌本身的核心定义与表达方式。
- **Build the System**：把品牌转化为可重复执行、可验证、可接入、可维护的系统。

Brand Guide 只是生命周期中的一种表达产物，不是 Skill 的起点，也不是最终目标。

---

## 2. Brand Lifecycle 总体范围

Brand System Builder 应覆盖以下九个生命周期领域：

1. Audit：审计
2. Foundation：品牌基础
3. Identity：品牌身份创建
4. Systemization：品牌系统化
5. Application：品牌应用
6. Validation：验证与治理
7. Integration：最终接入
8. Freeze：版本冻结
9. Maintenance：长期维护与演进

这些领域定义 Skill 必须理解和负责的生命周期边界，不代表固定、线性的执行流程。

---

## 3. Audit：审计

审计是 Brand System Builder 的正式起点。

审计对象可以是已有品牌，也可以是准备建立的新品牌。

### 对已有品牌

应覆盖：

- 品牌战略与实际表达是否一致
- 品牌名称、定位、价值主张是否清晰
- Logo、色彩、字体和视觉资产现状
- 网站、产品、文档、社交媒体等触点表现
- 品牌语言和内容表达
- 不同渠道之间的一致性
- 正式资产、历史资产和临时资产的区分
- 品牌资产的可用性、完整性和来源
- 设计与代码中的实际实现状态
- 已有规范是否仍然有效
- 规则、资产和实现之间是否发生漂移
- 授权、字体、商标和无障碍等风险提示

### 对新品牌

应覆盖：

- 已知业务背景
- 产品或服务边界
- 目标受众
- 市场与品类语境
- 已确认的品牌设想
- 创始人或团队偏好
- 已存在的名称、域名、产品资产
- 必须继承或必须避免的内容
- 建立品牌所需信息是否充分

### 审计范围的关键原则

审计不只是“收集文件”，而是建立品牌生命周期的初始基线：

- 当前拥有什么
- 哪些内容可信
- 哪些内容冲突
- 哪些内容缺失
- 哪些内容已经失效
- 哪些内容可以继承
- 哪些内容需要重建

---

## 4. Foundation：品牌基础

该阶段定义品牌为什么存在、面向谁、代表什么。

应覆盖：

- 品牌目的
- 品牌愿景
- 品牌使命
- 核心价值
- 目标受众
- 品牌定位
- 核心差异
- 价值主张
- 品牌承诺
- 品牌个性
- 品牌原则
- 品牌叙事
- 核心信息
- 品牌命名语义
- 品牌架构与产品关系
- 品牌不是什么

这一领域必须把抽象战略转化为可指导后续创作的决策基础。

### 边界

Brand System Builder 可以：

- 基于业务材料整理和形成品牌基础
- 发现定位矛盾
- 提出候选定义
- 将模糊想法变成明确选择
- 建立后续视觉和语言决策依据

但不替代：

- 大规模市场研究
- 专业用户调研
- 商业战略咨询
- 商标注册
- 法律尽职调查

如果没有足够证据，应将结论标记为假设或待验证方向。

---

## 5. Identity：品牌身份创建

这是“Brand Building”区别于“Brand Management”的核心范围。

Brand System Builder 应支持定义和创建品牌身份，而不只是整理已有资产。

### 视觉身份

应覆盖：

- Logo 概念与标志体系
- 主标、辅助标和符号关系
- 色彩方向
- 字体方向
- 排版特征
- 图形语言
- 形状语言
- 图标风格
- 插画风格
- 摄影风格
- 动效特征
- 空间感与构图原则
- 品牌识别性元素
- 视觉禁区

### 语言身份

应覆盖：

- 品牌声音
- 品牌语气
- 用词原则
- 信息层级
- 标语与核心表达
- 不同场景下的语气变化
- 正面与反面示例
- 禁止使用的表达
- AI 内容生成边界

### 体验身份

应覆盖：

- 品牌给人的整体感受
- 品牌在关键触点中的行为方式
- 视觉、语言与交互之间的一致性
- 品牌个性如何转化为实际体验
- 哪些体验特征必须稳定存在

### 边界

Brand System Builder 应负责：

- 建立身份方向
- 定义创作约束
- 生成或协调生成候选方案
- 比较候选方案
- 判断方案是否符合品牌基础
- 将批准方向发展为完整身份系统

它不应承诺替代专业设计师的所有原创能力，也不应把未经批准的 AI 生成结果直接认定为正式品牌资产。

---

## 6. Systemization：品牌系统化

品牌身份被确认后，需要从“设计结果”转化为“可重复运行的系统”。

应覆盖：

- 品牌概念与具体规则的映射
- Logo 体系及使用条件
- 色彩角色与组合逻辑
- 字体层级与排版规则
- 图形、图像、图标和动效规则
- 品牌语言规则
- 跨媒介适配原则
- Design Token
- 命名体系
- 资产分类
- 文件版本与状态
- 使用示例
- 禁止事项
- 允许变化的范围
- 不允许变化的核心
- 例外处理原则

系统化的目标不是制作一份更长的指南，而是回答：

- 哪些决策必须保持一致
- 哪些决策可以根据场景变化
- 如何判断一个产物是否属于该品牌
- 人、代码和 AI 如何使用同一套品牌规则

Brand Guide、Token、模板、参考资产和机器可读配置都只是品牌系统的不同载体。

---

## 7. Application：品牌应用

品牌只有在真实触点中成立，才算完成建设。

Brand System Builder 应覆盖品牌系统在典型场景中的应用定义：

- 网站
- Web 应用
- 移动应用
- 产品 UI
- 营销页面
- 社交媒体
- 演示文稿
- 文档
- 邮件
- 广告
- 印刷品
- 包装
- 活动材料
- 视频
- AI 生成内容
- 内部品牌材料

该领域应解决：

- 抽象品牌原则如何映射到具体触点
- 不同媒介如何保持同一品牌识别
- 哪些规则是全局规则
- 哪些规则只适用于特定场景
- 品牌资产是否能承受实际使用
- 是否存在只能在品牌板上成立、无法落地的设计

### 边界

Skill 可以建立应用规范、模板要求和示范基准。

它不应默认承担所有页面、视频、广告和印刷品的批量生产；具体产物可由其他专业 Skill 完成，但应接受品牌系统的约束。

---

## 8. Validation：验证与治理

品牌建设不能仅以“看起来不错”作为完成标准。

Brand System Builder 应覆盖：

- 战略与身份是否一致
- 视觉与语言是否一致
- 不同触点是否能够识别为同一品牌
- 系统规则是否足够明确
- 资产是否完整
- 关键场景是否得到覆盖
- Token 与规范是否一致
- 设计与实现是否一致
- 品牌是否具备基本可访问性
- 是否存在授权或合规风险
- 是否仍有未决事项
- 哪些偏差属于错误
- 哪些偏差属于合理例外
- 哪些结果需要人工批准

验证对象不只是 Brand Guide，而是完整链路：

> 品牌基础、品牌身份、系统规则、实际产物与最终实现是否彼此一致。

---

## 9. Integration：最终接入

Integration 指品牌系统正式进入生产和组织工作环境。

应覆盖的接入对象包括：

- 设计工具
- 代码仓库
- Design Token 系统
- 前端主题
- 组件库
- CMS
- 文档模板
- 演示文稿模板
- 内容生产系统
- AI Prompt 与品牌上下文
- DAM 或品牌资产库
- 团队工作规范
- 发布与审核机制

Brand System Builder 的职责是定义并验证：

- 哪些品牌资产需要接入
- 每项资产的正式来源
- 设计侧与代码侧如何对应
- Token 如何映射到具体平台
- 哪个系统是事实来源
- 如何避免重复维护
- 哪些旧资产需要停止使用
- 接入结果是否忠实于批准版本
- 下游系统是否能够稳定消费品牌规则

### 边界

Integration 属于 Skill 的生命周期范围，但不意味着它必须亲自完成所有平台操作。

它可以协调或调用其他能力完成接入；涉及生产代码、发布、外部系统写入或高风险变更时，仍需遵循相应工程与审批边界。

---

## 10. Freeze：版本冻结

Freeze 不是“以后不再修改”，而是建立一个可追溯的正式品牌基线。

应覆盖：

- 正式版本号
- 冻结日期
- 批准状态
- 批准主体
- 正式品牌基础
- 正式身份方案
- 正式资产清单
- 正式 Token
- 正式模板
- 正式实现映射
- 已知例外
- 已知限制
- 未纳入范围的内容
- 被替代的旧版本
- 资产来源与依赖
- 恢复或回滚基线

冻结后的品牌版本应具备：

- 唯一识别
- 明确边界
- 可复现
- 可验证
- 可回滚
- 可供下游系统稳定引用

没有 Freeze，品牌系统就会长期处于“最新版不知道在哪里”的状态。

---

## 11. Maintenance：长期维护与演进

品牌进入生产后，Skill 应支持持续维护，而不是将交付视为终点。

应覆盖：

- 新触点扩展
- 新资产纳入
- 品牌一致性审查
- 设计与代码漂移检测
- Token 变更管理
- 模板更新
- 过时资产识别
- 品牌例外管理
- 问题修复
- 小版本优化
- 大版本演进
- 品牌刷新
- 子品牌扩展
- 合并、拆分或退役
- 变更影响分析
- 兼容性判断
- 迁移要求
- 回滚要求
- 历史版本保留

Maintenance 必须区分：

- 不改变品牌核心的修复
- 扩展品牌能力的增量变化
- 改变品牌识别的重大升级
- 临时例外
- 应被拒绝的品牌漂移

品牌维护的目标不是阻止变化，而是让变化可控、可追踪，并保持核心连续性。

---

## 12. 横跨整个生命周期的共同职责

无论品牌处于哪个阶段，Brand System Builder 都应持续维护以下能力。

### 12.1 状态区分

所有品牌内容应能够区分为：

- 已确认事实
- 已批准决策
- 工作假设
- 候选方向
- 建议
- 待确认项
- 已冻结资产
- 已废弃资产

### 12.2 来源追踪

关键决策应能说明：

- 来源是什么
- 为什么作出该决策
- 当前是否仍然有效
- 哪些系统正在使用它

### 12.3 一致性

持续检查以下链路：

> 商业意图 → 品牌基础 → 品牌身份 → 系统规则 → 应用产物 → 技术实现

### 12.4 可逆性

重要品牌变更应考虑：

- 影响范围
- 旧版本兼容性
- 迁移方式
- 回滚方式
- 历史资产处理方式

### 12.5 人机共同使用

品牌系统既要能被人理解，也要能被：

- 设计工具
- 开发系统
- 内容工具
- 自动化流程
- AI Agent

稳定消费。

---

## 13. 生命周期之外的事情

以下事项与品牌有关，但不属于 Brand System Builder 的直接职责：

- 企业经营战略制定
- 完整市场研究
- 大规模用户研究
- 商标注册和法律结论
- 域名或公司注册
- 广告投放管理
- 社交媒体运营
- 公关活动执行
- 全量内容生产
- 全量 UI/UX 产品设计
- 生产系统的无授权变更
- 替代最终品牌决策者进行批准
- 保证生成资产不存在版权或商标风险

Skill 可以识别这些依赖、准备输入或定义交接要求，但不应吞并相邻专业领域。

---

## 14. 修订后的 Scope 定义

> **Brand System Builder 覆盖品牌从审计、基础定义、身份创建、系统化、实际应用、验证、生产接入、版本冻结到长期维护与演进的完整生命周期。它的核心不是生成 Brand Guide，而是建立一个能够被创建、应用、验证、集成、冻结和持续演进的品牌系统。**

其生命周期起点是：

> **理解当前品牌状态与建设条件。**

其阶段性完成标志是：

> **品牌已经进入真实系统，并形成可追溯、可复现、可回滚的冻结版本。**

其长期终点不是一次性交付，而是：

> **品牌在持续变化中仍保持识别连续性、规则一致性和系统可维护性。**
